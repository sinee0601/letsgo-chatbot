# 키워드 캐시(keyword_caches) 도입 계획

## Context (왜 하는가)
현재 `/chat` 요청은 매번 Gemini를 호출한다. 본 챗봇 질의는 모두 "레저스포츠 관광지"에 대한
것이라 동일·유사 질문이 반복될 가능성이 높다. `keyword_caches` 테이블에 **질문에서 뽑은 키워드 →
응답**을 저장해 두고, 같은 키워드 질의가 오면 Gemini 호출 없이 캐시 응답을 돌려줘 비용/지연을 줄인다.

결정사항 (사용자 확인):
- 키워드 추출: **규칙 기반 파싱** (LLM 호출 없음)
- 캐시 적중 시: **캐시 응답을 반환하되 `chat_logs`에도 정상 기록** (세션 이력 유지)

테이블 (init.sql에 이미 존재):
```sql
keyword_caches(
  keywords        VARCHAR(100) PK,
  cached_response TEXT NOT NULL,
  updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
)
```

## 변경 흐름 (router `chat()`)
```
1. keywords = extract_keywords(request.message)        # 규칙 기반
2. cached   = get_cached_response(db, keywords)          # keywords 비었으면 skip
3. 적중:  bot_response = cached,            model = "cache"
   미적중: history 조회 → generate_response() 호출,  model = settings.gemini_model
           keywords 있으면 upsert_cache(db, keywords, bot_response)
4. save_chat_log(...)  →  ChatResponse 반환  (적중/미적중 모두 동일)
```
- 응답 스키마(`ChatResponse`)는 그대로 `chat_logs` 한 건을 반환하므로 변경 없음.
- 캐시에서 온 로그는 `model="cache"`로 표기해 구분 (이력에서 식별 가능).

## 수정/추가 파일

### 1. `app/keywords.py` (신규) — 규칙 기반 추출기
순수 함수 `extract_keywords(user_message: str) -> str`:
- 소문자화 + `strip`, 특수문자 → 공백 치환
- 토큰별 끝 조사 제거(은/는/이/가/을/를/에서/에/의/로/으로/와/과/도/만 …)
- 불용어 제거(어때/알려줘/추천/어디/곳/장소/여행/관광지/정보 …)
- `sorted(set(tokens))`로 중복 제거 + 어순 무관 정규화 → `" ".join()[:100]` (VARCHAR(100) 보장)
- 결과가 빈 문자열이면 호출부에서 캐시를 건너뛴다(빈 키 저장 금지).
- 불용어/조사 목록은 모듈 상단 상수로 두어 튜닝 가능하게 한다.

> 참고: 사용자가 스케치한 `check_cached(user_message)`는 `contents=[]`(Gemini 패턴)였으나,
> 규칙 기반으로 결정되어 Gemini가 불필요하고 DB 세션 접근이 필요하므로,
> "추출(순수함수) + 조회/저장(repository)"으로 책임을 분리한다.

### 2. `app/models.py` — `KeywordCache` 모델 추가
```python
class KeywordCache(Base):
    __tablename__ = "keyword_caches"
    keywords: Mapped[str] = mapped_column(String(100), primary_key=True)
    cached_response: Mapped[str] = mapped_column(Text, nullable=False)
    updated_at: Mapped[datetime] = mapped_column(
        DateTime, nullable=False, default=datetime.utcnow, onupdate=datetime.utcnow
    )
```
- `main.py`의 `Base.metadata.create_all`이 자동 생성하므로 init.sql과 정합.

### 3. `app/repository.py` — 캐시 조회/저장 함수
- `get_cached_response(db, keywords) -> str | None`
  `select(KeywordCache.cached_response).where(...)` → `scalar_one_or_none()`
- `upsert_cache(db, keywords, response) -> None`
  MySQL `INSERT ... ON DUPLICATE KEY UPDATE` 사용
  (`from sqlalchemy.dialects.mysql import insert as mysql_insert`) →
  `on_duplicate_key_update(cached_response=stmt.inserted.cached_response)` 후 `commit()`
  (PK 충돌 시 응답 갱신, `updated_at`은 DB가 자동 갱신)

### 4. `app/router.py` — `chat()` 핸들러에 캐시 분기 추가
위 "변경 흐름"대로 구현. `extract_keywords` import, `settings` import,
`get_cached_response`/`upsert_cache` import 추가. 미적중 경로에서만 history 조회.

## 변경하지 않는 것
- `app/gemini.py` (`generate_response`는 그대로 사용)
- `app/schemas.py`, `app/database.py`, `app/config.py`
- `init.sql` (이미 `keyword_caches` 정의 존재)

## 검증 (end-to-end)
1. MySQL 기동, 앱 실행: `.venv\Scripts\python.exe -m uvicorn app.main:app --reload`
2. 동일 질문 2회 POST:
   ```
   POST /chat {"session_id":"t1","message":"속초 서핑 어때?"}
   POST /chat {"session_id":"t1","message":"서핑 속초 알려줘"}   # 어순/표현 달라도 같은 키
   ```
   - 1회차: `model` = 실제 모델명(gemini-...), Gemini 호출 발생
   - 2회차: `model` = `"cache"`, 응답이 1회차와 동일, Gemini 미호출
3. `SELECT * FROM keyword_caches;` → 정규화된 keywords 1행, `cached_response` 저장 확인
4. `GET /chat?session_id=t1` → 두 로그 모두 기록되어 세션 이력 유지 확인
5. 빈/특수문자만 있는 메시지로 keywords가 비는 경우 캐시를 건너뛰고 정상 동작하는지 확인
